## Workitem Documentation

### What was built

Two complementary changes that make memory recall return better results for vague or paraphrased queries — without adding any embedding model, API key, or new service:

1. **`/dako:recall` slash command** — now expands the user's query into up to 5 keyword variants (the original plus 1-4 paraphrases generated by the agent), calls the existing `recall` MCP tool once per variant, and merges the results by rank-based fusion before presenting them grouped by memory type.

2. **`## Memory Query Expansion` section in `CLAUDE.md`** — documents the expansion protocol so that any agent-initiated call to `recall` (LTM) or `find_patterns` (STM) follows the same approach, not just the user-invoked `/recall` command.

The MCP servers were not modified. No new dependencies. The cap of 5 variants lives in the skill and protocol text — schemas of `recall` and `find_patterns` are unchanged.

### How it works

**Variant generation.** The agent generates up to 5 total queries — the original plus 1-4 paraphrases. The skill provides concrete guidance: use synonyms ("save" ↔ "store" ↔ "persist" ↔ "remember"), vary specificity ("memory" ↔ "long-term memory" ↔ "MongoDB memories"), try inverse framings — but skip trivial morphological variations like singular/plural alone.

**Per-variant calls.** Each variant calls the existing MCP tool independently with the same `limit`. The `recall` MCP returns results ordered by MongoDB `$text` score internally, but the formatted output does not expose numeric scores. The skill therefore uses **rank-based fusion**: the score for each unique memory is the number of variants where it appeared, with ties broken by the best (lowest) rank across those variants.

**Per-tier dedup keys.** LTM uses `[TYPE] title` (the prefix the `recall` MCP emits in its formatted output, e.g. `[DECISION] Use MongoDB`) — this matches what the `forget` tool uses for deletion, so it is the natural stable key. STM uses a content fingerprint (first 80 characters of `content`, lowercased, whitespace-collapsed) because `find_patterns` does not emit a stable structured key.

**Fallback.** If a variant errors, it is skipped — remaining variants still produce results. If every variant returns nothing, behavior matches the pre-change `/recall`: report plainly, do not invent context.

**Presentation.** Merged results are sorted by score desc, then best rank asc. The skill caps presentation at 5-10 results to avoid noise from the multiplied result set (5 variants × limit 5 = up to 25 raw results before dedup).

### Usage

```
/recall <keywords>
```

Same command surface as before. The expansion happens automatically — users do not need to think about variants. The agent may optionally annotate variant coverage on each result (e.g. "matched 3/5 variants") to signal confidence.

For agent-initiated memory searches during normal work, follow `CLAUDE.md` § Memory Query Expansion: expand when the user's intent could be expressed in multiple ways, skip when the query is a specific function or file name.

### Known limitations

Review verdict was `pass` with no accepted gaps.

Two observations worth noting (design choices, not gaps):

- **Variant quality depends on the agent.** The same query may produce different variants on different runs. The smoke test demonstrated that even mediocre variants typically catch additional relevant memories — but if quality is visibly weak in practice, the skill text is the lever to tighten.
- **N round-trips per recall instead of 1.** At local-network MongoDB latency this is sub-second and not noticeable. If recall is ever moved behind a slow link, batching becomes worth considering.

The path to true semantic search (embedding-based, option D from the propose phase) is preserved as a backlog item: "Local embedding model for recall". This workitem deliberately did not pursue that path to keep the self-hosted ethos of DakoHarness intact (no API keys, no model downloads).
