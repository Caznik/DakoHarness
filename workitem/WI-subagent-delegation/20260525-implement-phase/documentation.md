## Workitem Documentation

### What was built

A new Claude Code custom sub-agent named **`wi-implementer`**, and a rewrite of the `/wi-implement` slash command to dispatch the implement phase to that sub-agent rather than running it inline in the main conversation.

Before this workitem, when you ran `/wi-implement`, all of the work — architecture review, exploring files with the Explore agent, writing tests, writing code, iterating on QA failures, running regression — happened in your main conversation. The main agent's context window absorbed every file read, every grep, every Edit, every test output. On a non-trivial workitem that's hundreds of tool calls. Once the context filled up, Claude Code would auto-compact, losing useful state and forcing the agent to re-derive context.

After this workitem, `/wi-implement` does only three things in main: resolve the workitem path, initialize `implementation.md`, and dispatch a sub-agent. The sub-agent — `wi-implementer` — runs in its own context window, does the full implement phase (5.1 architecture → 5.2 TDD coding → 5.3 QA loop with the new AC Pre-Check protocol → 5.4 regression), writes everything to `implementation.md`, and returns a terse status to main (`done -> <path>` plus 3–5 highlight bullets, or `blocked -> <reason>`, or `replan-requested -> <discovery>`).

The main agent reads `implementation.md` only if it needs detail. The user sees the same end result (a completed implement phase ready for `/wi-review`), but main's context burn drops dramatically.

Three patterns adapted from the user's prior-project agent set (preserved in `old-agents/`):
- **Scope discipline** — explicit "no unplanned refactors, no architecture invention, no scope creep" framing baked into the sub-agent's system prompt.
- **Failure conditions list** — explicit "STOP and return BLOCKED" triggers (plan ambiguous, AC ambiguous, architecture unclear, etc.), so the sub-agent escalates cleanly rather than guessing.
- **Replan Request Protocol** — when the sub-agent discovers the plan is *fundamentally* incompatible with the repository (not just a small deviation), it writes a structured Replan Request block to `implementation.md` and returns `replan-requested`. Main surfaces three options to the user (re-run `/wi-plan`, adjust scope manually, cancel). Max one replan per sub-feature.

One new artifact pattern: the **AC Pre-Check table** in `implementation.md`. Each AC from `analyze.md` gets a row with a `Test / Evidence` reference and a `COVERED` / `MISSING` status. The sub-agent must populate every row to `COVERED` before returning `done`. The skill's `done` handler in main sanity-checks the table — if any row is `MISSING`, the return is converted to `blocked` even if the sub-agent claimed success.

### How it works

**File layout (three-location mirror convention, parallel to skills):**

```
.claude/agents/wi-implementer.md        # dev mode — what Claude Code reads in this repo
agents/wi-implementer.md                # repo-root mirror (parallel to standalone commands/)
claude-plugin-release/agents/wi-implementer.md  # shipped with the plugin distribution
```

All three are byte-identical. The skill `/wi-implement` is mirrored the same way in `.claude/commands/`, `commands/`, and `claude-plugin-release/commands/`.

**Agent file format:**

```yaml
---
name: wi-implementer
description: <one-line description used in agent picker>
tools: Read, Edit, Write, Grep, Glob, Bash
---

# wi-implementer Sub-Agent
...
```

The frontmatter `tools` field restricts the agent's toolset — `wi-implementer` deliberately does NOT have the `Agent` tool, so it cannot spawn nested sub-agents. It does its own exploration with `Grep` and `Glob` instead of calling the Explore agent. This keeps the dispatch graph one level deep and easier to reason about.

The body is the agent's system prompt. It's organized as:

1. Role and core principle
2. Mandatory Inputs (refusal-to-start clause if missing)
3. Scope Discipline (forbidden behaviours, small vs. material deviation rule)
4. Failure Conditions (7 explicit STOP triggers → `blocked`)
5. Phase Protocol (5.1 Architecture / 5.2 TDD / 5.3 QA loop with AC Pre-Check / 5.4 Regression)
6. AC Pre-Check protocol (populate, flip MISSING→COVERED, never weaken an AC)
7. Replan Request Protocol (fundamental incompatibility → `replan-requested`)
8. Return Contract (the three valid return forms with templates)
9. Forbidden Actions (no chat narration, no nested Agent calls, no source_of_truth edits, no git writes)

**Skill dispatch flow:**

`/wi-implement` runs six steps:

1. Resolve the workitem path.
2. Initialize `<workitem>/implementation.md` with `status: in-progress` and an empty AC Pre-Check table seeded with one `MISSING` row per AC from `analyze.md`.
3. Check the replan attempt counter — read `source_of_truth.md` Key Decisions Log, count rows starting with `wi-implement dispatch #`. If ≥2 AND the previous dispatch was a replan-requested, refuse to dispatch and surface `blocked -> max replans exceeded`.
4. Invoke the `Agent` tool with `subagent_type: "wi-implementer"`. Prompt contents: workitem path, path to `implementation.md`, full text of `plan.md`, full text of `analyze.md`, one-line execution directive.
5. Parse the return. Three branches:
   - `done -> <path>` → sanity-check the AC Pre-Check (any `MISSING` row converts the return to `blocked`); print highlights; update `source_of_truth.md` (current phase + updated + sub-features).
   - `blocked -> <reason>` → read Blockers detail; append to `source_of_truth.md` Active Blockers; set `Blocked: yes`; surface to user.
   - `replan-requested -> <discovery>` → read Replan Request block; present three-option prompt (re-run `/wi-plan` / adjust scope / cancel).
6. Hand back to user — do NOT auto-chain into `/wi-review`. User runs `/wi-next`.

**Why agent-driven, not a hook or a tool wrapper:**

A SessionStart hook can't run inside Claude Code's tool call graph. A custom MCP tool can't open its own context window — anything it returns goes back into the main agent's window. The Agent tool, with a `.claude/agents/` definition, is the only mechanism that gives a real isolated context window for the dispatched work. Token cost: the sub-agent re-derives setup context once per dispatch, but everything after that runs in its window. Net win on any non-trivial workitem.

**Why max 1 replan:**

If `plan_v2` also fails, the problem is upstream — the analyze ACs or the user's understanding, not the plan. A third auto-attempt won't help; user judgement does. Same constraint the user's prior-project `implementer.md` had.

### Usage

No new command. You still run `/wi-implement <workitem path>` exactly as before. The dispatch happens automatically.

Three observable changes from the user's perspective:

1. **Less mid-phase chatter in main.** The sub-agent doesn't narrate its work in the conversation. You see one line confirming dispatch, then a terse return when it's done.
2. **AC Pre-Check shows up in `implementation.md`.** A new table between Blockers and QA Log maps each AC to a test/evidence reference and a `COVERED` / `MISSING` status. Useful when stepping through `/wi-review`.
3. **Replan path.** If the sub-agent discovers the plan is fundamentally wrong (vs. a small deviation it can adapt to), `/wi-implement` returns with three options instead of plowing through. You decide whether to re-plan, adjust scope, or cancel.

For everything else — phase ordering, `/wi-next` chaining, blocker handling, the `accepted-with-gaps` flow — behaviour is unchanged.

### Known limitations

**AC-10 (live runtime smoke test) was accepted as a gap during review.** The acceptance criterion called for a live dispatch of `/wi-implement` against a throwaway workitem to verify the full runtime path. The dispatch was attempted in the implementation session but failed with `Agent type 'wi-implementer' not found` — Claude Code's agent registry is populated at session start, so new files in `.claude/agents/` aren't invokable until you restart your session. The implementation fell back to static wiring inspection (Option B per the plan's Risk #3 mitigation), which confirms the dispatch code paths are correct but doesn't prove the agent file executes end-to-end.

Verification is deferred to the next session: the next real `/wi-implement` run (against any actual workitem, after restarting Claude Code) serves as the live smoke test. The most likely runtime failures (frontmatter format errors, missing tool permissions) would produce errors distinct from "not found", so confidence is reasonably high — but it's not proven yet.

**Session restart required after adding a new sub-agent.** This is now documented in `obsidian-docs/Slash Commands.md` Sub-agents section. Any time you add or modify a file in `.claude/agents/`, restart your Claude Code session before the new agent becomes invokable. Existing sessions will see `Agent type '<name>' not found` until restarted. Not a defect — just a Claude Code discovery rule.

**Scope of this workitem is implement-only.** Other phases (`analyze`, `plan`, `review`, `document`) continue to run inline in the main agent. Splitting QA into a separate sub-agent (the implementer ↔ QA loop pattern from the user's `old-agents/`) was deliberately deferred — a future sub-feature under this WI can add it without disturbing the v1 dispatcher.
